# Follow-ups

Polish gaps surfaced during Supervisor review that did not block the PR they were flagged in. Each bullet is a concrete, user-facing gap the Supervisor can prioritize for a later phase or a small follow-up PR.

- **Investigate AGENT facet visual disambiguation.** Two agents with the same `agent_name` but different `client_type` (e.g. one `claude_code` plugin, one `flightdeck_sensor` SDK, both running as `omria@Omri-PC`) render as two identical-looking rows in the Investigate sidebar's AGENT facet list. The `client_type` pill is rendered in the main session table and the Fleet sidebar, but not in the facet row. Two choices for the fix: append a small uppercase pill (`CC` / `SDK`) next to the agent name inside the facet, or merge the two rows into one with a combined session count. (From phase 1.)

- **Agent-level search click routing uses `agent_name` as a `flavor=` filter value.** `buildSearchResultHref("agent", item)` in `dashboard/src/App.tsx` routes to `/investigate?flavor=<agent_name>` because the D115 `SearchResultAgent` no longer carries `agent_id`. For sensor-keyed agents whose `agent_name` is `user@hostname` this produces an empty session list on the Investigate page (no session has a flavor matching that value). The clean fix is to add `agent_id` to `store.SearchResultAgent` (+ swagger artifacts) and route to `/investigate?agent_id=<uuid>` so the existing D115 filter handles the lookup. (From phase 1.)

- **~~Agent-chip label fragility when the filter result is empty.~~** Closed by the Bug 2b fix landed on `fix/fleet-investigate-filters-ordering-polish` (second commit pair), and hardened in v0.4.0 Phase 2 by `GET /v1/agents/{id}`. The resolver chain is now: fleet-store `agents[]` → sessions list → `/v1/agents/{id}` cache (backed by `lib/agent-identity-cache.ts`) → 8-char UUID prefix + `console.warn`. With the by-id endpoint online the cache fills even when both the fleet store and the sessions list miss — the common repro being a direct `/investigate?agent_id=…` deep-link whose time window excludes every session the agent has. Listed here struck through for the historical trace of why it was filed and closed.

- **Phase 4 dashboard rendering polish.** The contract-level dashboard work for Phase 4 landed (new ``embeddings`` / ``llm_error`` event types rendered with badge + colour + filter-pill, per-theme ``--event-embeddings`` / ``--event-error`` tokens, typed ``LLMErrorPayload`` + ``StreamingMetrics`` shapes, and the ``error_type`` filter param wired through ``/v1/sessions``), but the richer drawer / Investigate surfaces were scoped out so the phase could ship. Follow-ups: (a) SessionDrawer event row for ``llm_error`` with an ``ErrorEventDetails`` accordion showing ``request_id`` + ``retry_after`` + ``is_retryable``; (b) a per-row STREAM badge + TTFT indicator on streaming ``post_call`` events; (c) Investigate ERROR TYPE facet (client-side facet + URL state) consuming the new API filter; (d) a small red dot next to the STATE column in the Investigate session table when a session has any ``llm_error`` event. Tracked here so a dashboard-polish PR can take them one at a time.

- **Bucket-crossing slide animation on the Fleet list.** Supervisor brief for Issue 3 asked for a 300 ms slide when an agent crosses a bucket boundary (LIVE / RECENT / IDLE). The bucket sort + stability / divider rendering landed in this PR; the animation did not. Adding CSS transitions on DOM rows that change parent position under React's key-stable re-render is fiddly — naive attempts either double-render or lose the leaving animation. Real fix needs a Framer-Motion-style layout animation primitive (`<LayoutGroup>` + `layout` prop) or a custom FLIP animation helper. Scope-deferred so the functional fix did not wait on the motion library decision.

- **~~`agents.total_sessions` rollup drift.~~** Closed by `POST /v1/admin/reconcile-agents` in PR #26. Admin-only endpoint recomputes `total_sessions`, `total_tokens`, `first_seen_at`, and `last_seen_at` from the sessions table on demand. Chose the on-demand-endpoint option (not a scheduled worker loop, not a trigger on `sessions`) per the Phase 3-bis V-pass decision lock — simpler to reason about and rare-enough that a scheduled loop would waste cycles. Orphan-row cleanup and multi-replica cross-process mutual exclusion explicitly out of scope. See `docs/admin-endpoints.md` for operator usage. Listed here struck through for the historical trace of why it was filed and closed.
