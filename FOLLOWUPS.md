# Follow-ups

Polish gaps surfaced during Supervisor review that did not block the PR they were flagged in. Each bullet is a concrete, user-facing gap the Supervisor can prioritize for a later phase or a small follow-up PR.

Three categories valid here: (a) gaps that fit the current phase's intent but were explicitly deferred via Supervisor decision in this PR, (b) post-launch follow-ups that fit a future Roadmap section, (c) closed items kept (struck through) for the historical trace. - **Investigate AGENT facet visual disambiguation.** Two agents with the same `agent_name` but different `client_type` (e.g. one `claude_code` plugin, one `flightdeck_sensor` SDK, both running as `omria@Omri-PC`) render as two identical-looking rows in the Investigate sidebar's AGENT facet list. The `client_type` pill is rendered in the main session table and the Fleet sidebar, but not in the facet row. Two choices for the fix: append a small uppercase pill (`CC` / `SDK`) next to the agent name inside the facet, or merge the two rows into one with a combined session count. (From phase 1.)

- **Agent-level search click routing uses `agent_name` as a `flavor=` filter value.** `buildSearchResultHref("agent", item)` in `dashboard/src/App.tsx` routes to `/investigate?flavor=<agent_name>` because the D115 `SearchResultAgent` no longer carries `agent_id`. For sensor-keyed agents whose `agent_name` is `user@hostname` this produces an empty session list on the Investigate page (no session has a flavor matching that value). The clean fix is to add `agent_id` to `store.SearchResultAgent` (+ swagger artifacts) and route to `/investigate?agent_id=<uuid>` so the existing D115 filter handles the lookup. (From phase 1.)

- **~~Agent-chip label fragility when the filter result is empty.~~** Closed by the Bug 2b fix landed on `fix/fleet-investigate-filters-ordering-polish` (second commit pair), and hardened in v0.4.0 Phase 2 by `GET /v1/agents/{id}`. The resolver chain is now: fleet-store `agents[]` → sessions list → `/v1/agents/{id}` cache (backed by `lib/agent-identity-cache.ts`) → 8-char UUID prefix + `console.warn`. With the by-id endpoint online the cache fills even when both the fleet store and the sessions list miss — the common repro being a direct `/investigate?agent_id=…` deep-link whose time window excludes every session the agent has. Listed here struck through for the historical trace of why it was filed and closed.

- **~~Embedding-modality content capture parity.~~** Closed by PR #28 (S-EMBED-1..8). Pre-fix the sensor emitted ``event_type=embeddings`` with metadata only; the request's ``input`` parameter was never captured even with ``capture_prompts=True``. Communication-modality parity gap caught during Chrome walkthrough. Fix shipped: ``PromptContent.input: str | list[str] | None`` round-trips through ``payload.content.input`` → ``event_content.input`` JSONB → ``GET /v1/events/:id/content`` → ``<EmbeddingsContentViewer>``. Coverage: OpenAI direct, litellm direct, LangChain transitive via OpenAI; smoke + integration + vitest + T14 E2E. Anthropic / Claude Code N/A. Voyage-direct via LangChain remains deferred (Q-VOYAGE decision).

- **~~Framework attribution always-null.~~** Closed by PR #28 (parallel finding). Pre-fix ``Session.record_framework`` had zero callers across the sensor codebase, so every event silently emitted ``framework=null`` — the dashboard FRAMEWORK facet, analytics ``group_by=framework``, and ``/v1/sessions?framework=`` filter were all silently misbehaving. Fix shipped: sensor wires ``record_framework`` at ``init()`` from ``FrameworkCollector`` with bare-name strip; LangChain classifier accepts ``langchain_core`` alias; ``/v1/sessions?framework=`` SQL OR-combines bare-name and JSONB-array. Listed here struck through for the historical trace of why a Phase 1 oversight was load-bearing for Phase 4.

- **~~Swimlane expanded-drawer dead-end on agents whose only sessions are > 7 days old.~~** Closed by PR #28 (V-DRAWER fix). Pre-fix ``loadExpandedSessions`` called ``fetchSessions(agent_id, limit=100)`` with no ``from`` bound, so the server's 7-day default narrowed the result and an agent with only ancient sessions read "No sessions to display for this agent." in the drawer. Fix shipped: ``from = new Date(0).toISOString()`` returns all-time sessions; real pagination via ``EXPANDED_DRAWER_PAGE_SIZE = 25`` + "Show older sessions" load-more; "View in Investigate →" footer deep-link. Companion fix: ``buildFlavors`` filters out ``total_sessions=0`` orphan agents from the swimlane. T5b regression guard spec covers the post-fix contract.

- **Bucket-crossing slide animation on the Fleet list.** Supervisor brief for Issue 3 asked for a 300 ms slide when an agent crosses a bucket boundary (LIVE / RECENT / IDLE). The bucket sort + stability / divider rendering landed in this PR; the animation did not. Adding CSS transitions on DOM rows that change parent position under React's key-stable re-render is fiddly — naive attempts either double-render or lose the leaving animation. Real fix needs a Framer-Motion-style layout animation primitive (`<LayoutGroup>` + `layout` prop) or a custom FLIP animation helper. Scope-deferred so the functional fix did not wait on the motion library decision.

- **~~`agents.total_sessions` rollup drift.~~** Closed by `POST /v1/admin/reconcile-agents` in PR #26. Admin-only endpoint recomputes `total_sessions`, `total_tokens`, `first_seen_at`, and `last_seen_at` from the sessions table on demand. Chose the on-demand-endpoint option (not a scheduled worker loop, not a trigger on `sessions`) per the Phase 3-bis V-pass decision lock — simpler to reason about and rare-enough that a scheduled loop would waste cycles. Orphan-row cleanup and multi-replica cross-process mutual exclusion explicitly out of scope. See `docs/admin-endpoints.md` for operator usage. Listed here struck through for the historical trace of why it was filed and closed.

- **Orphan-agent reconciler.** ``agents`` rows accumulate from prior test runs, smoke fixtures, and one-off integrations and never get cleaned up. The dev DB carries 130+ such rows. Phase 4 polish (V-DRAWER fix) filters ``total_sessions=0`` agents from the Fleet swimlane as a UX workaround, but the rows still occupy the AGENT facet, the ``/v1/agents`` listing, and the agent-id resolver chain. A scheduled reconciler (sibling to ``POST /v1/admin/reconcile-agents``) that nulls or hard-deletes agent rows whose sessions are all closed and older than a configurable threshold would close the loop. Out of scope for this PR because the workaround unblocked the V-DRAWER fix; queued for a small follow-up PR or a dedicated admin endpoint (``POST /v1/admin/reap-orphan-agents``) per Supervisor preference.
