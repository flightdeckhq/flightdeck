# Phase 1 Follow-ups

Polish gaps surfaced during Supervisor review that did not block the phase 1 merge. Each bullet is a concrete, user-facing gap the Supervisor can prioritize for phase 3 or a post-merge follow-up PR.

- **Investigate AGENT facet visual disambiguation.** Two agents with the same `agent_name` but different `client_type` (e.g. one `claude_code` plugin, one `flightdeck_sensor` SDK, both running as `omria@Omri-PC`) render as two identical-looking rows in the Investigate sidebar's AGENT facet list. The `client_type` pill is rendered in the main session table and the Fleet sidebar, but not in the facet row. Two choices for the fix: append a small uppercase pill (`CC` / `SDK`) next to the agent name inside the facet, or merge the two rows into one with a combined session count. Scope-in for phase 3 polish or a small follow-up PR; not a regression.

- **Agent-level search click routing uses `agent_name` as a `flavor=` filter value.** `buildSearchResultHref("agent", item)` in `dashboard/src/App.tsx` routes to `/investigate?flavor=<agent_name>` because the D115 `SearchResultAgent` no longer carries `agent_id`. For sensor-keyed agents whose `agent_name` is `user@hostname` this produces an empty session list on the Investigate page (no session has a flavor matching that value). The clean fix is to add `agent_id` to `store.SearchResultAgent` (+ swagger artifacts) and route to `/investigate?agent_id=<uuid>` so the existing D115 filter handles the lookup. Left out of the phase 1 close-out because the scope was the broken search endpoint, not the UX it surfaces.
