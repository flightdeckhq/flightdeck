# Follow-ups

Polish gaps surfaced during Supervisor review that did not block the PR they were flagged in. Each bullet is a concrete, user-facing gap the Supervisor can prioritize for a later phase or a small follow-up PR.

- **Investigate AGENT facet visual disambiguation.** Two agents with the same `agent_name` but different `client_type` (e.g. one `claude_code` plugin, one `flightdeck_sensor` SDK, both running as `omria@Omri-PC`) render as two identical-looking rows in the Investigate sidebar's AGENT facet list. The `client_type` pill is rendered in the main session table and the Fleet sidebar, but not in the facet row. Two choices for the fix: append a small uppercase pill (`CC` / `SDK`) next to the agent name inside the facet, or merge the two rows into one with a combined session count. (From phase 1.)

- **Agent-level search click routing uses `agent_name` as a `flavor=` filter value.** `buildSearchResultHref("agent", item)` in `dashboard/src/App.tsx` routes to `/investigate?flavor=<agent_name>` because the D115 `SearchResultAgent` no longer carries `agent_id`. For sensor-keyed agents whose `agent_name` is `user@hostname` this produces an empty session list on the Investigate page (no session has a flavor matching that value). The clean fix is to add `agent_id` to `store.SearchResultAgent` (+ swagger artifacts) and route to `/investigate?agent_id=<uuid>` so the existing D115 filter handles the lookup. (From phase 1.)

- **~~Agent-chip label fragility when the filter result is empty.~~** Closed by the Bug 2b fix landed on `fix/fleet-investigate-filters-ordering-polish` (second commit pair). `buildActiveFilters` now resolves the chip label via the fleet store's `agents[]` first, the sessions list second, the UUID prefix only as a final fallback; Investigate hydrates the fleet store on mount so the lookup has real data even on a direct-navigation deep link. A module-level `warnUnresolvedAgentOnce` emits one `console.warn` per agent_id that still falls through to the UUID prefix, so future operators have a signal to file `GET /v1/agents/{id}` if it hurts. Listed here struck through for the historical trace of why it was filed and closed.

- **Bucket-crossing slide animation on the Fleet list.** Supervisor brief for Issue 3 asked for a 300 ms slide when an agent crosses a bucket boundary (LIVE / RECENT / IDLE). The bucket sort + stability / divider rendering landed in this PR; the animation did not. Adding CSS transitions on DOM rows that change parent position under React's key-stable re-render is fiddly — naive attempts either double-render or lose the leaving animation. Real fix needs a Framer-Motion-style layout animation primitive (`<LayoutGroup>` + `layout` prop) or a custom FLIP animation helper. Scope-deferred so the functional fix did not wait on the motion library decision.
